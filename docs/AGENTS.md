# AGENTS.md

## 目的

このリポジトリは **神社・寺院コンシェルジュシステムのバックエンド** を管理します。

コンシェルジュ API を通じて、安定した・説明可能な神社レコメンデーションを提供することが目標です。
フロントエンドが使用する API の契約（レスポンス形式）は厳守してください。

このドキュメントは、このリポジトリで作業する **人間・AI エージェント**（Codex、ChatGPT、自動化スクリプト等）向けの開発ルールを定義します。

---

## コア開発原則

1. **API レスポンスの契約を絶対に壊さない**
   フロントエンドが使用するフィールドは、明示的に承認されない限り変更禁止。

2. **変更は小さく・独立して行う**
   大規模なリファクタリングは、特性テストを先に追加してから実施する。

3. **バグ修正には必ずリグレッションテストを追加する**

4. **変更前後でテストが通ることを確認する**

5. **ランキング動作・レコメンデーションロジックをこっそり変更しない**

6. **暗黙的な動作より明示的なロジックを優先する**

7. **迷ったらロジックを変えずにテストを追加する**

## ランキングロジック

スコアは **表示用** と **内部ランキング用** の2種類があり、意図的に分離されています。

| スコア | フィールド | 用途 |
|---|---|---|
| API 契約スコア | `breakdown.score_total` | 表示・説明用 |
| 内部ランキングスコア | `rec["_score_total"]` | 内部ソート用 |

> **注意:** これら2つのスコアを統合・マージしてはいけません。テストがこの区別に依存しています。

## 候補パイプラインのルール

候補生成処理は以下のファイルで行われます：

```
temples/services/concierge_chat_candidates.py
```

以下を必ず守ること：

1. 座標のない神社は除外する
2. 候補には `distance_m` と `place_id` を含める（利用可能な場合）
3. 候補は `popular_score DESC` 順で並べる
4. 重複除去の優先順位：
   - `place_id`
   - `shrine_id / id`
   - `name + address`
5. ユーザー指定の候補はビルト候補より先に保持する

## リクエスト正規化のルール

コンシェルジュビューでリクエストの正規化を行います。

**対象ファイル:** `temples/api_views_concierge.py`

### filters のプロモーション

`filters` 内の以下フィールドはトップレベルに昇格できます：

- `birthdate`
- `goriyaku_tag_ids`
- `extra_condition`

### トップレベルの優先

同じフィールドがトップレベルと `filters` 両方に存在する場合、**トップレベルの値が優先**されます。

### フロー検出

以下のいずれかが存在する場合、**フロー B** が起動します：

- `goriyaku_tag_ids` が存在する
- `extra_condition` が存在する

---

## フォールバック動作

フィルタリング結果がゼロ件の場合、フォールバックモードが起動します。

| 項目 | 期待動作 |
|---|---|
| `result_state.fallback_mode` | `true` になる |
| レコメンデーション順 | 距離順でソート |
| 返答メッセージ | フォールバックの免責文を含む |

## テストのルール

### テスト実行順序

```bash
# Step 1 — 対象テストを実行
pytest <test_file> -q -vv

# Step 2 — 全テストスイートを実行
pytest -q --maxfail=1

# Step 3 — 失敗があれば修正してから続行
```

> **失敗したテストをコミットしてはいけません。**

### テスト設計の方針

**優先するテスト:**

- API 契約テスト
- ランキング契約テスト
- フォールバック動作テスト
- 重複除去テスト
- リクエスト正規化テスト

**避けるべきテスト:**

- 壊れやすいスナップショットテスト
- 内部実装の詳細テスト
- ログ出力のテスト

---

## AI エージェントのワークフロー

タスクを実行する際は、以下の順序で作業すること：

1. 関連ファイルを読む
2. 既存のテストを理解する
3. カバーされていないケースにテストを追加する
4. 必要な場合のみコードを変更する
5. テストを実行する
6. 結果を報告し変更内容をまとめる

> **投機的なリファクタリングは行わないこと。**
